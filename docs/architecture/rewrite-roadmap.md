# 重寫路線圖

## 目標

以保留既有資料庫業務價值為前提，重寫應用層、模組邊界與測試體系，最終取代 `D:\rental` 舊系統。

## 決策原則

- 先重建應用結構，不先追著舊 bug 補
- 先明文化資料契約，再做資料遷移或相容層
- 先完成可垂直切片的模組，再逐步替換舊流程

## 路線分期

### Phase 0: 盤點與契約固定

- 產出資料契約
- 釐清狀態機
- 確認舊資料中的異常樣式
- 決定哪些欄位保留、哪些欄位淘汰

完成標準：

- `data_contracts/` 下有核心表契約文件
- `reports/` 下有欄位與流程對照

### Phase 1: 新版骨架

- 建立 `app/` 分層結構
- 定義 app factory、config、db session、錯誤處理、logging
- 建立模組邊界與測試框架

完成標準：

- 可啟動空白新版 app
- 模組目錄與 README 完整

### Phase 2: 核心主資料模組

- `auth`
- `landlords / properties / rooms`
- `tenants / contracts`

完成標準：

- 可完成登入與主資料 CRUD
- 主資料查詢不再依賴舊單檔 `app.py`

### Phase 3: 帳單與計費

- `billing`
- `electricity`
- `water`

完成標準：

- 月帳單可生成
- 電費與水費流程有可驗證 service
- 計算公式有單元測試

### Phase 4: 付款與報表

- `payments`
- `reports`
- 對帳、月報、房東報表

完成標準：

- 付款流程只有一套正式模型與狀態機
- 報表查詢與頁面對應一致

### Phase 5: 整合與切換

- LINE / OCR 整合
- 資料遷移腳本
- 回歸測試
- 上線切換計畫

完成標準：

- 舊版與新版可對帳
- 有切換與回滾步驟

## 不可跳過的驗收關卡

- 關卡 A：資料契約凍結
- 關卡 B：骨架與模組邊界確認
- 關卡 C：帳單/電水/付款流程測試通過
- 關卡 D：報表對帳通過
- 關卡 E：切換方案通過
